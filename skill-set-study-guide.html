REST APIs Very likely On your resume
⌄
What it is
REST API is a way for two applications to talk to each other over the internet using standard web requests. Think of it like a menu at a restaurant: the API defines what you can ask for (GET, POST, PUT, DELETE), and the server brings back what you ordered in a structured format (usually JSON). REST stands for Representational State Transfer.
Key concepts to know
  • HTTP verbs: GET = read data, POST = create something new, PUT/PATCH = update, DELETE = remove
  • Stateless: Each request is completely independent. The server doesn't "remember" you from the last request. Any session info must come with the request itself (like a JWT token).
  • Status codes: 200 = success, 201 = created, 400 = bad request (your fault), 401 = not authenticated, 403 = no permission, 404 = not found, 500 = server error
  • JSON: The standard data format sent back and forth. Looks like: {"name": "Vicki", "role": "engineer"}
  • Endpoints: The specific URL paths your API exposes, like /api/users or /api/events/123
What to mention (PaceCircle project)
I built a production-grade REST API for PaceCircle, a social fitness platform. I designed endpoints for circle management, workout event scheduling, and role-based ownership. I used Spring Boot and followed REST conventions throughout: GET for retrieving data, POST for creating circles and events, PUT for updates, and DELETE for removals. I also implemented a global exception handler using @RestControllerAdvice so error responses were consistent and meaningful across all endpoints.
If they go deeper
They may ask the difference between REST and GraphQL. REST has fixed endpoints per resource; GraphQL lets the client ask for exactly the fields it needs in one request. The Cisco team uses REST, so you won't need to defend GraphQL, but knowing the tradeoff is a plus.
Role-Based Access Control (RBAC) Very likely On your resume
⌄
What it is
RBAC is a security system where what you're allowed to do in an application is determined by your role, not your individual identity. For example: an "admin" can delete records, a "manager" can edit them, and an "employee" can only view their own. It's how you prevent someone from accessing data or actions they shouldn't touch.
Key concepts
  • Roles: Labeled permission groups (admin, manager, viewer, owner)
  • Permissions: The actual actions allowed per role (read, write, delete)
  • Enforcement points: You check roles at the API layer (backend), not just the UI. A bad actor can bypass buttons in the frontend and call your API directly.
  • Cisco's real use case: Their Senior AI Engineer built "an app for processing employee data with role-based access, enabling managers and admins to alter employment history and employees to make elections." This is textbook RBAC.
What to mention (two angles)
From PaceCircle: I implemented role-based ownership enforcement in the backend where circle owners have different permissions than regular members. From Bank of America: I performed access reviews and database maintenance on 120+ development and production datasets, ensuring 160+ developers only had access to the data their role required. Both taught me that access control needs to live at the data layer, not just the UI.

JWT authentication Very likely On your resume
⌄
What it is
JWT stands for JSON Web Token. It's a compact, self-contained way to verify who you are when making API requests. After you log in, the server gives you a signed token. You attach this token to every future request, and the server verifies the signature to confirm it's legitimate without needing to look you up in a database each time. It has three parts: a header (algorithm used), a payload (your user data/claims), and a signature (proves it hasn't been tampered with).
Why it matters to this team

The Cisco team builds HR tools used by 40,000+ employees. Authentication is critical. Their apps need to know who you are and what your role is before letting you touch employment data.

What to mention (PaceCircle)
In PaceCircle I implemented JWT authentication with a STATELESS session policy. That means the server never stores session data at all. Every request the client makes must include the JWT in the Authorization header. The server validates the signature and reads the claims (like user ID and role) directly from the token. This approach scales well because any server instance can validate any request without needing shared session storage.
Testing: unit tests and integration tests Likely On your resume
⌄
The difference
A unit test checks one small piece of code in isolation (like a single method) with no real database or network. An integration test checks that multiple pieces work correctly together, like your API endpoint + your service layer + your database. Unit tests are fast and pinpoint where a bug is. Integration tests are slower but confirm the whole chain works.
  • Mockito: A Java library for unit testing. It lets you "mock" (fake) dependencies so you can test one piece in isolation, e.g. fake the database layer so you're only testing your service logic.
  • MockMvc: A Spring testing tool that simulates HTTP requests to your controllers without starting a real server. Used for integration-style tests of your API layer.
  • Flyway: A database migration tool. It manages versioned SQL scripts so your database schema is always in sync with your code. Like version control for your database structure.
What to mention
In PaceCircle I'm writing Mockito unit tests for my service layer, mocking out repository dependencies so each test is isolated and fast. I'm also writing MockMvc integration tests to test the full request/response cycle for my API endpoints. I set up Flyway with 6 migration scripts to manage my PostgreSQL schema versions, so database structure changes are tracked and reproducible in any environment.
RAG (Retrieval-Augmented Generation) Very likely
⌄
What it is (plain English)
AI models are trained on public internet data up to a certain date. They don't know anything about your company's internal documents or database. RAG solves this. Before asking the AI a question, you first search your own documents or database for relevant information, then include that information in the prompt you send the AI. The AI uses both its general knowledge AND your specific retrieved data to give a better, more accurate answer. Think of it as giving the AI a cheat sheet pulled from your own files.
How it works step by step
  • User asks a question
  • System searches a knowledge base (database, documents, emails) for relevant chunks
  • Those chunks are inserted into the prompt alongside the question
  • The AI generates an answer using both the retrieved content and its own training
Why this team cares deeply

The Senior AI Engineer built "an AI application to auto-populate requests in an IT system from employee and external emails via RAG." Pulling context from emails, then having the AI understand and act on them, is exactly what RAG is for.

What to mention (Android Story Generator)
My Android Story Generator app used a similar pattern. I integrated the Google Cloud Vision API to extract structured data from images (automated tagging), then passed those tags as context to the TextCortex API for story generation. I was essentially augmenting the generation step with retrieved/extracted information. I'd love to apply this to a proper RAG system, which is something I've been studying since it's central to building useful enterprise AI tools.
Key term: vector database
They may mention this. A vector database stores documents as numerical representations (embeddings) that capture meaning rather than exact words. When you search it, you find semantically similar content, not just keyword matches. Examples: Pinecone, Weaviate, pgvector (a PostgreSQL extension). You don't need deep experience, just awareness of what they are and why they're used in RAG.

What it is
A regular AI just answers a question. An AI agent can take actions. You give it a set of tools (functions it can call, like "search the HR database" or "send a Slack message" or "update a ticket"), and the AI decides which tools to use and in what order to complete a goal. The agent reasons through a multi-step plan, calls tools, observes results, and keeps going until the task is done. This is what makes it an "agent" instead of a simple chatbot.

  • Orchestration: The layer of code that runs the loop: call LLM → parse tool call → execute tool → pass result back to LLM → repeat until done.
  • P&C Assistant: People & Culture (HR) assistant. An AI agent that helps employees with HR-related questions and tasks.

LLMs and prompt engineering Very likely
⌄
What LLMs are
Large Language Models (LLMs) are the AI models that generate text: GPT, Gemini, Llama, etc. They were trained on enormous amounts of text and learned statistical patterns of language. They predict the most likely next word given what came before. They don't "understand" in a human sense but they're remarkably capable at language tasks.
Their specific LLMs
Meta Llama (open source, can self-host) Google Gemini (commercial) VertexAI (Google's cloud AI platform)
Prompt engineering basics
  • System prompt: Instructions given to the model at the start that define its persona, rules, and context. Users don't usually see this.
  • Few-shot prompting: Giving the model examples of what you want before asking it to do the task.
  • Temperature: Controls randomness. Low (near 0) = more predictable and factual. High (near 1) = more creative. HR tools usually want low temperature.
  • Context window: The maximum amount of text an LLM can "see" at once. Everything has to fit: instructions, retrieved documents, conversation history, and the user's question.
What to mention
I was a two-time finalist in BofA's Agile Innovation Challenge, focusing on AI integration in documentation in 2024 and prompt engineering in 2025. The prompt engineering project taught me how significantly the structure and wording of a prompt affects model output, and how to design prompts that produce consistent, structured results for use in automated pipelines. That work gave me real appreciation for why enterprises need systematic prompt design rather than ad-hoc queries.

Database design and PostgreSQL Very likely On your resume
⌄
Key concepts
A relational database (like PostgreSQL) stores data in tables with rows and columns, like spreadsheets that can reference each other. SQL is the language used to query them. PostgreSQL is an open-source relational database and one of the most popular choices for production applications.
  • Schema: The structure/blueprint of your database: what tables exist, what columns, what data types, what constraints.
  • Foreign key: A column that references the primary key of another table, creating a relationship between them.
  • Indexing: Speeds up queries on specific columns. Without an index, a query scans every row. With one, it jumps directly to matching rows.
  • Flyway migration scripts: Numbered SQL files that Flyway runs in order to build or update your schema. Any developer can recreate the exact same database state from scratch.
  • Joins: Combining data from two or more tables. INNER JOIN = only rows that match in both. LEFT JOIN = all rows from the left plus matching rows from the right.

Docker and containers Likely On your resume
⌄
What it is
Docker packages an application and everything it needs to run (code, runtime, libraries, config) into a container — a standardized unit that runs the same way anywhere. The classic developer problem is "it works on my machine." Docker solves that: if it runs in the container on your laptop, it runs the same way in production. OpenShift (what BofA uses) runs Docker containers at scale.
  • Image: The blueprint for a container, like a class in OOP
  • Container: A running instance of an image, like an object
  • Dockerfile: A script that defines how to build the image (what OS, what dependencies, what code to copy in, what command to run)
  • Registry: A place to store and share images (like Docker Hub, or internal registries at enterprise companies)
What to mention

Caching and performance Likely
⌄
What caching is
Caching stores a copy of frequently requested data in fast-access memory so you don't have to recompute or re-fetch it from a slow source every time. Imagine asking a librarian for the same book 100 times. Instead of going to the shelf each time, they keep it on their desk. That's caching. The Cisco team's Senior AI Engineer specifically mentioned "improving user data access and response times through effective caching."
  • Cache hit: The data is already in cache, return it immediately
  • Cache miss: Data isn't in cache, fetch from source and store it
  • Cache invalidation: When the underlying data changes, you need to update or clear the cached version so you're not serving stale data. This is the hard part.
  • Redis: The most common caching tool. An in-memory key-value store. Extremely fast.
  • TTL (Time to Live): How long a cached item stays before it's automatically removed/refreshed
What to mention
I haven't implemented Redis caching yet, but I understand the tradeoffs clearly: you gain speed at the cost of potentially serving stale data. The right TTL and invalidation strategy depends on how frequently the underlying data changes. For an HR system, employee data changes infrequently enough that caching profile information makes sense, but real-time things like availability should always hit the source.
Apache Kafka (event-driven systems) Likely
⌄
What it is
Kafka is a messaging system. Instead of Service A calling Service B directly (tight coupling), Service A publishes an "event" to a Kafka topic (like posting to a channel), and any services that care subscribe and process it independently. This decouples your services. If Service B is down, the event waits in Kafka until B is back. Think of Kafka as a very fast, very durable message queue or event log.
  • Topic: A named channel for a category of events (e.g. "employee-created", "survey-submitted")
  • Producer: The service that publishes events
  • Consumer: The service(s) that subscribe and react to events
  • Why HR systems use it: When an employee submits a form, you might need to update 4 different systems. Kafka lets you do that asynchronously without them all needing to be available at the same instant.
What to mention if asked
I haven't used Kafka in production yet, but I understand the problem it solves. In a synchronous system, if one downstream service is slow or down, the whole chain stalls. Kafka decouples that by letting services communicate through durable, ordered event logs. For an HR system processing thousands of employee events, that reliability and scalability matters a lot. It's something I'm actively learning.
Application security fundamentals Very likely On your resume
⌄
Why this team cares
The Cisco team does "vulnerability scans and patches on applications used by 40,000+ users." That's a huge attack surface. HR data is especially sensitive (salaries, health info, personal details). Security is not optional for them.
Core vulnerabilities to know (OWASP Top 10)
  • SQL injection: An attacker puts SQL code into a form field to manipulate your database. Prevention: use parameterized queries or prepared statements, never build SQL strings by concatenating user input.
  • XSS (Cross-Site Scripting): Injecting malicious JavaScript into a webpage that other users see. Prevention: escape/sanitize output, use Content Security Policy headers.
  • Broken access control: Users can access things they shouldn't. Prevention: enforce authorization at the backend on every request, not just the frontend.
  • Sensitive data exposure: Storing or transmitting sensitive data without encryption. Prevention: HTTPS everywhere, hash passwords (bcrypt), never log sensitive fields.
  • Vulnerability scan: An automated tool that checks your application or infrastructure for known security weaknesses.
What to mention (two angles)
At BofA I performed access reviews ensuring confidential information was properly restricted for 160+ developers across 120+ datasets. That's directly about broken access control at scale. In PaceCircle I implemented JWT with a STATELESS session policy and a custom exception hierarchy. And in my Private Messaging project I went deep on cryptography, implementing the Double Ratchet algorithm with HKDF, AES-GCM, and curve P-256 to match the security standards used by Signal and WhatsApp.
Cryptography (your Private Messaging project) Likely On your resume
⌄
Key terms defined simply
This project is unusually impressive for a SWE 2 and worth being able to explain clearly without getting lost in jargon.
  • End-to-end encryption (E2EE): Only the sender and recipient can read the message. Even the server can't. The keys never leave users' devices.
  • Double Ratchet algorithm: The protocol Signal and WhatsApp use. It generates a new encryption key for every single message. Even if one key is compromised, past and future messages stay safe. ("Forward secrecy" + "break-in recovery.")
  • AES-GCM: A symmetric encryption algorithm. AES is the cipher (industry standard, very strong). GCM mode also provides authentication, meaning you can detect if the message was tampered with.
  • HKDF with SHA256: A key derivation function. Takes input key material and derives strong cryptographic keys from it in a secure, standardized way.
  • CCA-secure El-Gamal: CCA = Chosen Ciphertext Attack resistance. El-Gamal is a public-key encryption scheme. CCA-secure means it can't be broken even if the attacker can ask the system to decrypt chosen ciphertexts.
  • Your unique feature: Message reporting with CCA-secure encryption. You can report a message to a moderator without the moderator being able to read it unless they decide to act. Privacy-preserving moderation.
What to mention
I built an end-to-end encrypted chat client implementing the Double Ratchet algorithm — the same protocol Signal and WhatsApp use. Every message uses a freshly derived key, providing forward secrecy. The most interesting design challenge was building a message reporting feature: users can flag content for moderation, but the moderator can't see the content unless they actively decrypt it. I used CCA-secure El-Gamal encryption to enable this, so privacy is preserved by default even in the reporting flow. This project taught me how to reason about threat models, not just use crypto libraries as black boxes.
