OOP Fundamentals
Polymorphism · Encapsulation · Inheritance · Interfaces vs Abstract Classes
Polymorphism
watch for: compile-time vs runtime

The same method call behaves differently depending on the runtime type of the object. Enables writing generic code against abstractions rather than concrete types.

abstract class Animal { abstract String speak(); } class Dog extends Animal { String speak() { return "Woof"; } } class Cat extends Animal { String speak() { return "Meow"; } } Animal a = new Dog(); System.out.println(a.speak()); // "Woof" — resolved at runtime
Panel angle
Compile-time (static) polymorphism = method overloading — same name, different parameter types, resolved at compile time. Runtime (dynamic) polymorphism = method overriding via inheritance — resolved by JVM dispatch table at runtime. Interviewers love this distinction.
Encapsulation

Bundle data and behavior together, and restrict direct access to internal state via access modifiers: private, protected, public.

public class BankAccount { private double balance; // hidden from outside public void deposit(double amount) { if (amount > 0) balance += amount; // guarded write } public double getBalance() { return balance; } }

The benefit is invariant protection — callers can't set balance to a negative number directly. All mutation paths go through your logic.

Inheritance

A subclass extends a superclass, inheriting fields and methods. Use when a clear "is-a" relationship exists. Java supports single-inheritance for classes only.

class Vehicle { int speed; void accelerate() { speed += 10; } } class Car extends Vehicle { int doors; // inherits speed + accelerate() }
Watch out
Deep inheritance chains become rigid and hard to refactor. Favor composition over inheritance when the relationship is "has-a" rather than "is-a". E.g. a Car has an Engine — don't extend Engine.
Interface vs Abstract Class
most common OOP interview question
Interface

Pure contract. Since Java 8 can have default and static methods. No instance state. A class can implement many interfaces.

interface Flyable { void fly(); default String mode() { return "air"; } }
Abstract Class

Can have instance fields, constructors, and concrete methods. Single inheritance only. Use when subclasses share common state or base logic.

abstract class Shape { String color; // shared state abstract double area(); void describe() { System.out.println(color); } }
Use Interface whenMultiple unrelated classes share a capability (Serializable, Comparable, Runnable). Prefer for API design. Supports multiple inheritance of type.
Use Abstract Class whenSubclasses share state or partial implementation. You need a constructor, initializer logic, or non-public members.

HashMap internals · ArrayList vs LinkedList · When to use each
HashMap internals
frequently asked

HashMap stores entries in an array of buckets. Each key is hashed via hashCode(), the hash determines the bucket index. Collisions within a bucket are stored as a linked list — and since Java 8, that list converts to a red-black tree when a bucket hits 8 entries.

// What happens under the hood for map.put("hello", 42): // 1. hash = "hello".hashCode() → e.g. 99162322 // 2. index = hash & (capacity - 1) → e.g. bucket 2 (capacity 16) // 3. Store Entry{key="hello", value=42} at buckets[2] // 4. Collision → chain as linked list → tree at bucket size 8
Key numbersDefault capacity: 16
Load factor: 0.75
Rehash when: size > capacity × 0.75
Rehash cost: O(n) — doubles capacity
hashCode contractIf a.equals(b) then a.hashCode() == b.hashCode() must hold. Violate this and HashMap lookups silently break — you store a value but can never retrieve it.
Panel angle
If asked about worst-case HashMap performance: without the Java 8 tree optimization it was O(n) per get due to long collision chains. With treeification it degrades to O(log n) in the worst case — but average is O(1).
ArrayList vs LinkedList
ArrayListBacked by a dynamic array. O(1) random access by index. O(n) insert/delete in middle (shifts elements). Low memory overhead. CPU cache-friendly. Default choice for most lists.
LinkedListDoubly-linked nodes. O(1) insert/delete at head or tail. O(n) random access. High memory overhead (two pointers per node). Implements Deque — useful as a queue.
// Use ArrayList for almost everything List<String> list = new ArrayList<>(); // Use LinkedList only as a queue/deque Deque<String> queue = new LinkedList<>(); queue.offer("first"); queue.poll(); // removes and returns "first" // Actually: ArrayDeque is faster than LinkedList as a deque Deque<String> better = new ArrayDeque<>();
Panel tip
If asked "when would you use LinkedList?" — answer: as a queue or deque implementation, not as a general-purpose list. ArrayList wins in almost every benchmark due to memory locality. And for a deque, ArrayDeque is faster than LinkedList anyway.

Module 03
Streams & Lambdas
filter · map · collect · Optional · Lambda syntax
Streams overview
Java 8+

Streams let you process sequences of elements in a declarative pipeline. They are lazy — intermediate operations don't execute until a terminal operation is called. Streams are not data structures; they don't store data.

List<String> names = List.of("Alice", "Bob", "Anna", "Charlie"); List<String> result = names.stream() .filter(n -> n.startsWith("A")) // intermediate, lazy .map(String::toUpperCase) // intermediate, lazy .collect(Collectors.toList()); // terminal — triggers pipeline // result: ["ALICE", "ANNA"]
filter, map, collect
filter — Predicate<T> → keeps elements where predicate is true
stream.filter(x -> x > 0) // keep positives stream.filter(s -> s.length() > 3) // keep long strings
map — Function<T,R> → transforms each element
stream.map(String::length) // Stream<String> → Stream<Integer> stream.mapToInt(String::length).sum() // primitive stream, avoids boxing
collect — terminal, accumulates results
stream.collect(Collectors.toList()) stream.collect(Collectors.toSet()) stream.collect(Collectors.joining(", ")) // groupingBy — produces Map<K, List<V>> stream.collect(Collectors.groupingBy(String::length)) // {3: ["Bob"], 5: ["Alice", "Anna"], 7: ["Charlie"]}
Optional
null safety

A container that may or may not hold a value. Forces the caller to handle the "empty" case rather than risking a NullPointerException silently.

Optional<String> opt = Optional.of("hello"); opt.isPresent() // check before get opt.get() // throws NoSuchElementException if empty opt.orElse("default") // fallback value opt.orElseGet(() -> computeFallback()) // lazy fallback opt.orElseThrow(RuntimeException::new) opt.map(String::toUpperCase) // transform if present opt.ifPresent(System.out::println) // side effect
DoUse Optional for method return types to signal "might not have a result". Chain with map/flatMap for clean transformations.
Don'tUse Optional as method parameters or as fields. Don't call get() without isPresent() check. Don't wrap things that are never null.
Lambda syntax & functional interfaces
// Lambda vs method reference — equivalent list.forEach(s -> System.out.println(s)); list.forEach(System.out::println); // cleaner // Core functional interfaces Predicate<T> fn = t -> true/false; // filter Function<T,R> fn = t -> r; // map Consumer<T> fn = t -> {}; // forEach Supplier<T> fn = () -> t; // orElseGet, lazy factory BiFunction<T,U,R> fn = (t, u) -> r; // two-arg transform
Streams vs for-loops — tradeoffs
Streams winReadability for multi-step transformations. parallelStream() for CPU-bound work on large data. Composability — chain operations without temp variables.
For-loop winsEasier to debug — no lambda stack frames. Better performance for tight loops — streams have overhead. Early break/continue logic that's awkward with streams.
Module 04
Concurrency Basics
synchronized · volatile · ExecutorService
synchronized
mutual exclusion

A synchronized method or block acquires an intrinsic lock (monitor) on an object. Only one thread can hold that lock at a time — preventing race conditions on shared mutable state.

class Counter { private int count = 0; public synchronized void increment() { count++; // only one thread executes this at a time } // equivalent — explicit block on a specific lock public void increment() { synchronized (this) { count++; } } }
Cost & alternative
Threads block waiting for the lock — contention under load is a bottleneck. For a simple counter, prefer AtomicInteger, which is lock-free and uses hardware CAS (compare-and-swap) instructions.
volatile

volatile guarantees visibility — a write by one thread is immediately visible to all other threads, bypassing CPU cache. It does not guarantee atomicity of compound operations like i++ (which is read-modify-write).

class Worker { private volatile boolean running = true; // visible to all threads public void stop() { running = false; } public void run() { while (running) { // reads fresh value, not cached doWork(); } } }
Use volatileSimple status flags written by one thread, read by many. Single write operations where atomicity isn't needed.
Don't use volatileRead-modify-write operations (i++). Multiple variables that must be updated atomically. Use synchronized or Atomic* instead.
ExecutorService
thread pool management

Managing raw threads is error-prone and doesn't scale. ExecutorService provides a managed thread pool — you submit tasks, the pool handles thread lifecycle.

ExecutorService pool = Executors.newFixedThreadPool(4); // Runnable — fire and forget pool.submit(() -> System.out.println("task")); // Callable — returns a result Future<Integer> future = pool.submit(() -> { return expensiveComputation(); }); int result = future.get(); // blocks until done // Always shut down cleanly pool.shutdown(); pool.awaitTermination(5, TimeUnit.SECONDS);
Factory methodWhen to use
newFixedThreadPool(n)CPU-bound tasks — set n = CPU cores
newCachedThreadPool()Many short-lived tasks — threads expire after 60s idle
newSingleThreadExecutor()Sequential task queue — guaranteed order
newScheduledThreadPool(n)Delayed or periodic tasks
newVirtualThreadPerTaskExecutor()Java 21 — lightweight virtual threads for I/O-bound work
synchronized vs volatile vs Atomic — decision matrix
synchronizedFull mutual exclusion. Multiple shared variables. Compound operations. Highest overhead.
volatileVisibility only. Single variable. One-writer / many-readers. Zero locking overhead.
Atomic (AtomicInteger, AtomicReference…)Lock-free, uses CAS. Best for counters and single-variable state shared across threads. Better throughput than synchronized under low-to-medium contention. Backed by java.util.concurrent.atomic.
Also worth knowing
ConcurrentHashMap — thread-safe map with bucket-level locking, much faster than synchronized HashMap. BlockingQueue — classic producer-consumer pattern. CountDownLatch — wait for N threads to finish. ReentrantLock — explicit lock with tryLock() and fairness options.
